# Action-to-Artifact Binding

## Purpose

Action-to-Artifact Binding records how a specific action contributes to an artifact or to a defined fragment of an artifact.

The basic structure is:

```text
Action
   ↓
Transformation
   ↓
Artifact
```

In more complex workflows:

```text
Root Intent
     ↓
Delegation
     ↓
Agent Action
     ↓
Intermediate Output
     ↓
Transformation
     ↓
Human Selection
     ↓
Human Edit
     ↓
Final Artifact
```

The purpose of v0.3 is not merely to show that an artifact exists.

It is to preserve the production lineage connecting actions, transformations, and artifact regions.

## Why Artifact-Level Attribution Is Insufficient

Complex artifacts are rarely produced by one isolated action.

A report may contain:

* research performed by one agent;
* analysis performed by another agent;
* calculations produced by a tool;
* paragraphs generated by a language model;
* sections reorganized by a human;
* conclusions selected and rewritten by a human reviewer.

A simple record such as:

```text
Agent A → Report X
```

may therefore hide significant causal complexity.

Action-to-Artifact Binding allows a record to identify a more precise scope:

```text
Agent A
   ↓
Report X
   └── Chapter 3
```

or:

```text
Tool B
   ↓
Dataset X
   └── /results/forecast
```

or:

```text
Human C
   ↓
Source File
   └── Lines 40–92
```

## Binding Model

A binding record contains:

* an action reference;
* an artifact reference;
* a binding type;
* a contribution scope;
* optional transformation steps;
* supporting evidence;
* binding confidence;
* verification status.

Conceptually:

```text
Action Reference
       ↓
Binding Type
       ↓
Contribution Scope
       ↓
Artifact Reference
```

## Binding Types

### `generated`

The action produced new artifact content.

### `transformed`

The action materially transformed an existing input into a new output.

### `assembled`

The action combined multiple existing components.

### `edited`

The action modified an existing artifact or artifact fragment.

### `selected`

The action selected an output from alternatives.

Selection is treated as a meaningful contribution event.

### `verified`

The action checked, validated, or confirmed an artifact.

Verification does not imply authorship of the artifact itself.

### `annotated`

The action added metadata, comments, labels, or explanatory material.

### `rejected`

The action rejected a candidate artifact or fragment.

Rejection may be important in decision lineage even when the rejected artifact does not appear in the final output.

### `other`

The relationship does not fit the predefined binding types.

## Contribution Scope

The `contribution_scope` field identifies what part of an artifact the action is bound to.

Supported scope types include:

* whole artifact;
* section;
* file;
* JSON Pointer;
* line range;
* byte range;
* time range;
* custom scope.

Example:

```yaml
contribution_scope:
  scope_type: section
  locator: "chapter-3"
```

Another example:

```yaml
contribution_scope:
  scope_type: line_range
  locator: "src/main.py"
  start: 40
  end: 92
  unit: line
```

The protocol deliberately separates artifact identity from contribution scope.

One artifact may therefore have multiple Action-to-Artifact Bindings.

## Transformation Chains

A direct binding is not always sufficient.

For example:

```text
Research Trace
      ↓
Agent Analysis
      ↓
Summary
      ↓
Human Selection
      ↓
Human Rewrite
      ↓
Published Section
```

The optional `transformation_chain` records intermediate transformations.

A transformation step identifies:

* transformation type;
* input references;
* output reference;
* actor;
* timestamp;
* optional notes.

The goal is not to store the entire workflow payload.

The goal is to preserve enough references to reconstruct the material transformation path.

## Relationship to Delegation Causality Chain

v0.2 records responsibility propagation.

v0.3 records production binding.

The difference is:

```text
v0.2:
Who was asked to do what?

v0.3:
What did an action contribute to?
```

Combined:

```text
Root Intent
     ↓
Delegation Chain
     ↓
Delegated Action
     ↓
Action Receipt
     ↓
Artifact Binding
     ↓
Artifact Fragment
```

An action reference may therefore contain:

* an Action Receipt reference;
* a Delegation Chain reference;
* a Delegation Step reference.

This preserves the path from originating intent to artifact production.

## Selection as Contribution

The protocol treats selection as distinct from generation.

Consider:

```text
Agent generates 100 candidates
           ↓
Human reviews candidates
           ↓
Human selects one candidate
           ↓
Human modifies selected candidate
           ↓
Final artifact
```

The agent may have a `generated` binding.

The human may have:

* a `selected` binding;
* an `edited` binding.

This allows later contribution analysis to distinguish different kinds of participation.

v0.3 does not assign numerical economic value to these contribution types.

That task belongs to later contribution assessment and royalty systems.

## Verification Without Authorship

An agent or human may verify an artifact without producing it.

For example:

```text
Agent A → generated code
Agent B → verified code
Human C → approved release
```

These are different relationships.

The protocol records them separately.

Verification should not automatically become authorship attribution.

## Evidence

A binding requires evidence.

Possible evidence includes:

* Action Receipts;
* Delegation Records;
* Transformation Logs;
* version-control commits;
* artifact diffs;
* execution logs;
* human testimony;
* external sources.

Evidence does not eliminate uncertainty.

The `binding_confidence` field makes the assessment level explicit.

## Artifact Integrity

Artifact references may include a content hash.

The purpose is to help identify the exact artifact version to which a binding applies.

A binding to one version of an artifact should not automatically be assumed to apply to every later version.

## Non-Goals of v0.3

Version 0.3 does not attempt to:

* calculate royalty percentages;
* determine legal authorship;
* resolve copyright disputes;
* assign ownership;
* prove plagiarism;
* calculate contribution value;
* replace task protocols;
* replace tool execution protocols;
* replace version-control systems.

Its role is narrower:

> Bind actions and transformations to artifacts or artifact fragments through evidence-backed records.

## Relationship to Future Versions

The protocol develops as follows:

```text
v0.1
Causality Link Record
        ↓
v0.2
Delegation Causality Chain
        ↓
v0.3
Action-to-Artifact Binding
        ↓
v0.4
Contribution Causality Graph
        ↓
v0.5
Unified Causality Lifecycle
```

v0.3 creates the bridge required for v0.4.

Before a system asks:

> How much did each participant contribute?

it should first be able to ask:

> Which actions are connected to which parts of the artifact?

The protocol therefore places evidence before valuation.
